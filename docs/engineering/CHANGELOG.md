## 1. 目的

本 CHANGELOG 用于记录 B1 Platform 项目全生命周期中的重要工程里程碑、架构决策、技术变更与版本演进。

### CHANGELOG 与 Git Commit 的区别

| 维度 | CHANGELOG | Git Commit |
|------|-----------|------------|
| **粒度** | 面向版本/冲刺级别，聚合多个提交 | 单次代码变更，粒度细 |
| **受众** | 开发团队、竞赛评委、维护者 | 开发者、代码审查者 |
| **语言** | 面向人类阅读的结构化中文说明 | 技术性描述，可混杂中英文 |
| **目的** | 追溯工程里程碑、支持版本发布说明 | 版本控制、代码追溯、协作 |
| **维护方式** | 每个 Sprint 结束后手动编写 | 每次提交自动记录 |

CHANGELOG 是对 Git 历史的提炼与升华，反映"做了什么、为什么做"而不仅是"改了什么"。

## 2. 版本号规则

本项目遵循 **语义化版本（Semantic Versioning，SemVer）** 规范：

```
v<MAJOR>.<MINOR>.<PATCH>
```

| 位 | 名称 | 说明 | 典型触发条件 |
|----|------|------|-------------|
| MAJOR | 主版本号 | 不兼容的 API 变更 | 架构重构、核心模块推翻重做、破坏性接口变更 |
| MINOR | 次版本号 | 向下兼容的功能新增 | 新增模块/页面/功能、Mock → 真实接口替换 |
| PATCH | 修订号 | 向下兼容的问题修复 | Bug 修复、样式微调、文案修正、性能优化 |

**示例：**

- `v0.1.0` — 初始开发版本，含基础框架与核心页面
- `v0.2.0` — 新增检查评价模块
- `v1.0.0` — 竞赛发布正式版

> **注意：** 在项目进入正式发布（v1.0.0）之前，所有版本均为 `0.x.x` 开发阶段，API 稳定性不保证。

## 3. 变更分类标准

每次版本发布时，按照以下标准标签对变更进行分类：

| 标签 | 含义 | 示例 |
|------|------|------|
| **Added** | 新增功能、模块、页面 | 新增学生自评模块 |
| **Changed** | 已有功能的修改或调整 | 修改评分规则由百分制改为等级制 |
| **Improved** | 功能增强、体验优化 | 优化表格加载性能 |
| **Fixed** | Bug 修复 | 修复导出报表中文乱码问题 |
| **Removed** | 移除的功能或代码 | 移除旧版评分接口 |
| **Deprecated** | 计划移除但暂保留的功能 | 标记旧版 API 为弃用 |
| **Security** | 安全性相关变更 | 修复 XSS 漏洞、添加 Token 校验 |
| **Documentation** | 文档变更 | 更新 API 接口文档 |
| **Refactor** | 代码重构（无功能变化） | 组件拆分、状态管理重构 |
| **Performance** | 性能优化 | 首屏加载时间优化 40% |

## 5. 变更日志模板

以下为未来版本发布时参考的模板，开发者在每个 Sprint 结束后按此格式补充新版本条目：

```markdown
## v<MAJOR>.<MINOR>.<PATCH> (YYYY-MM-DD)

> **发布说明：** 一句话概述本次发布的核心内容。

### Added

- 新增功能模块/页面描述

### Changed

- 已有功能的调整说明

### Improved

- 功能增强或体验优化描述

### Fixed

- Bug 修复描述

### Removed

- 移除的功能或代码说明

### Deprecated

- 标记为弃用的功能说明

### Security

- 安全性相关变更说明

### Documentation

- 新增或更新的文档列表

### Refactor

- 代码重构说明（功能无变化）

### Performance

- 性能优化指标说明
```

> **分类不是必填项。** 若某个版本无对应类型的变更，可省略该子标题。

## 7. 维护规则

### 更新时机

- **每个 Sprint 结束后**，由 Sprint 负责人更新本文件
- **每次版本发布前**，由 Tech Lead 审核并确认条目完整

### 更新步骤

1. 在本文档"初始版本"条目之上追加新的版本条目
2. 按 [变更分类标准](#3-变更分类标准) 对本次变更进行分类
3. 更新版本号，遵循 [语义化版本规则](#2-版本号规则)
4. 若有里程碑完成，同步更新 [开发里程碑时间线](#6-开发里程碑时间线) 中的状态
5. 提交 PR 并注明 `docs: update CHANGELOG for Sprint N`

### 编写原则

- **面向读者写：** 让不熟悉代码细节的人能理解变更的业务含义
- **一变更一行：** 每个条目聚焦一个变更点
- **过去时态：** 使用"新增了""修复了"描述已完成的工作
- **分类优先：** 先确定属于哪个变更分类，再组织句子
- **避免技术黑话：** 使用通用术语，必要时加注解释

### 审核清单

在合并 CHANGELOG 更新前，请检查：

- [ ] 版本号是否符合 SemVer 规范
- [ ] 所有条目均已正确分类
- [ ] 无遗漏的重要变更
- [ ] 里程碑时间线已同步更新
- [ ] 文档无拼写或格式错误
